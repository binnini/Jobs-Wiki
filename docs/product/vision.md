# Product Vision

## Problem

- 취업 시장의 핵심 문제는 정보 부족보다 정보 과잉과 파편화에 가깝습니다.
- 구직자는 고용24, 사람인 등 여러 플랫폼에 흩어진 정보를 직접 탐색하고 비교해야 합니다.
- 특히 신입 구직자는 자신의 역량과 채용 공고의 연결 지점을 찾기 어려워 선택 피로를 겪습니다.
- 고용24 Open API는 신뢰도 높은 데이터를 제공하지만, 이를 사용자 맥락에 맞게 연결한 서비스는 아직 부족합니다.
- 따라서 분산된 취업 정보를 사용자별 지식 베이스로 통합하고, LLM이 해석 가능한 구조로 재구성할 필요가 있습니다.

## Product Shape

- Jobs-Wiki는 단순 채용 공고 검색 서비스가 아니라 사용자별 취업 지식 워크스페이스를 지향합니다.
- 사용자는 같은 knowledge space를 파일 탐색, 문서 보기, 그래프, 캘린더, 검색 같은 여러 projection으로 탐색할 수 있어야 합니다.
- 사용자 수정 요청과 질의 요청은 웹 UI에서 시작할 수 있지만, 실제 mutation은 external command boundary를 통해 위임되는 구조를 전제로 합니다.

## Users

- 구직자
- 채용/직무 탐색 사용자
- 사용자-facing 웹 워크스페이스를 운영하는 내부 팀
- WAS 공개 API를 소비하는 외부 소비자(MCP 포함)

## Outcome

- 사용자는 단순 채용 공고 검색이 아니라 자신의 취업 지식 공간을 탐색할 수 있어야 합니다.
- 사용자는 문서, 링크, 일정, 그래프를 통해 취업 준비 상태를 하나의 워크스페이스에서 확인할 수 있어야 합니다.
- 사용자는 tree, document, graph, calendar, search 같은 서로 다른 화면에서 같은 지식 공간을 일관되게 이해할 수 있어야 합니다.
- 사용자는 이력서, 관심 분야, 활동 기록과 연결된 공고/교육/역량 정보를 더 빠르게 이해할 수 있어야 합니다.
- Jobs-Wiki는 사용자가 웹 환경에서 Obsidian 같은 PKM 도구처럼 취업 정보를 누적, 탐색, 활용할 수 있게 해야 합니다.
- Jobs-Wiki는 command success와 read visibility를 분리해, 동기화 지연이 있더라도 사용자에게 정직한 상태를 보여줄 수 있어야 합니다.

## Non-Goals

- MCP 서버 구현
- external knowledge backend 또는 DB ownership
- WAS 내부에서 ingestion orchestration 수행
- 단일 채용 포털 검색 화면만 제공하는 제품으로 축소하는 것
