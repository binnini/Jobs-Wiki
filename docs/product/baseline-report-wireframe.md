# Baseline Report Wireframe

## Purpose

이 문서는 Jobs-Wiki의 첫 진입 경험인 `기본 리포트` 화면을 저해상도 wireframe으로 정리합니다.

이 wireframe의 목적은 아래를 검토하는 것입니다.

- 사용자가 처음 30초 안에 무엇을 이해하는지
- 어떤 정보가 가장 먼저 보여야 하는지
- 어떤 다음 행동으로 이어져야 하는지

이 문서는 visual style을 확정하지 않습니다.
우선은 layout, hierarchy, navigation flow를 검토하는 것이 목적입니다.

## Primary UX Goal

사용자는 onboarding에서 프로필이나 문서를 입력한 직후,
아래 세 가지를 바로 이해해야 합니다.

- 시스템이 나를 어떻게 이해했는가
- 지금 어떤 공고와 신호가 중요한가
- 내가 다음에 무엇을 하면 되는가

## Desktop Reading Order

사용자가 위에서 아래로 자연스럽게 읽도록 아래 순서를 의도합니다.

1. Hero
2. Personal Snapshot
3. Recommended Opportunities
4. Action Queue
5. Market Brief / Skills Gap
6. Calendar / Ask Follow-ups

핵심 원칙:

- `추천 공고`가 가장 큰 메인 블록이어야 합니다.
- `개인 요약`은 신뢰 형성용이고, `액션 큐`는 행동 전환용입니다.
- `시장 브리핑`은 보조 설명이지만, 서비스의 차별성을 드러내는 영역입니다.

## Interaction Notes

### 1. Personal Snapshot

- onboarding 직후 사용자의 입력을 시스템이 어떻게 이해했는지 보여줍니다.
- 이 블록에서 틀린 정보가 있으면 사용자는 바로 수정할 수 있어야 합니다.

### 2. Recommended Opportunities

- 첫 화면의 메인 콘텐츠입니다.
- 사용자는 여기서 실제 행동 가능성을 느껴야 합니다.
- 각 카드에는 항상 `왜 관련 있는가` 한 줄이 있어야 합니다.

### 3. Action Queue

- 리포트를 읽고 끝나는 것이 아니라, 다음 행동으로 이어지게 만듭니다.
- 추천 개수는 많아도 3개를 넘기지 않는 편이 좋습니다.

### 4. Market Brief

- 개인화 정보와 별도로 `shared market intelligence`를 보여줍니다.
- 이 블록 덕분에 서비스가 단순 추천기가 아니라 분석 도구처럼 느껴집니다.

### 5. Ask Follow-ups

- 사용자가 빈 입력창 앞에서 막히지 않도록, 리포트에서 바로 자연스러운 질문을 제안합니다.

## Suggested Navigation Model

baseline report를 첫 홈으로 쓴다면, 1차 정보 구조는 아래처럼 두는 것이 자연스럽습니다.

- `Report`
  - 개인화된 기본 브리핑
- `Ask`
  - follow-up 질의
- `Opportunities`
  - 리스트 / 필터 / 상세 탐색
- `Calendar`
  - 마감 / 오픈 일정
- `Workspace`
  - 문서, 노트, 저장된 결과

## Feedback Points

이 wireframe을 검토할 때 우선 보면 좋은 질문은 아래입니다.

1. 첫 화면에서 `추천 공고`가 메인인 것이 맞는지
2. `시장 브리핑`의 비중이 지금보다 커야 하는지
3. `Action Queue`가 너무 도구적이거나 반대로 부족하지는 않은지
4. `Ask`를 별도 탭으로 두는 것이 자연스러운지
5. 첫 화면에서 `Workspace` 성격이 더 강하게 보여야 하는지

## Current Recommendation

현재 제 추천은 아래와 같습니다.

- 첫 화면은 `리포트형 홈`으로 둡니다.
- 메인 콘텐츠는 `추천 공고`로 둡니다.
- 차별화 포인트는 `시장 브리핑`과 `why matched` 설명에서 만듭니다.
- `Ask`는 기본 리포트를 확장하는 행동으로 둡니다.
- `Workspace`는 1차 메인이 아니라 후속 관리 공간으로 둡니다.
